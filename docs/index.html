// how_it_works

Not just a linter.
A full pipeline.

Most dead code tools stop at detection — they hand you a list of 500 warnings and wish you luck. Dead Code Archaeologist goes all the way: scan, score, delete, test, and open a PR.

01
AST scan
Tree-sitter parses every function definition and every function call across all 9 supported languages, building a complete call graph of your project.
02
Git analysis
Every file is checked against git history — when it was last touched, how many authors ever committed to it, and how many total commits it has received.
03
Confidence scoring
Five signals combine into a 0–100 score. Only functions scoring above your chosen threshold are flagged — reducing false positives dramatically.
04
Surgical deletion
Functions are removed by precise byte ranges from the AST — including decorators and annotations — leaving no orphaned blank lines or broken syntax.
05
Tests run first
Your test suite runs automatically. If anything fails, the branch is deleted and your codebase is restored exactly as it was. Nothing is committed unless tests pass.
06
PR opened
A GitHub PR is opened with a full description of every deleted function, its file location, confidence score, and the reasons it was flagged.
// commands_reference

Every command,
explained fully.

Click any command to expand the full documentation — every flag, what it does, when to use it, and what to expect.

python3 main.py [path] Scan a project and print a ranked table of dead code suspects ›
python3 main.py ./my-project --explain --min-confidence 50
This is the read-only scan command. It never modifies any files. It scans your entire project, builds a call graph, analyzes git history, scores every function, and prints a ranked table directly in your terminal. Use this first to understand what the tool finds before deciding to delete anything.

The output shows a confidence bar (filled blocks = higher confidence), the function name, the file and line number, the verdict label, and — with --explain — a plain-English reason for why it was flagged.
--explainShows a human-readable reason next to each result — e.g. "No callers found · Last touched 14 months ago · Only 1 author". Always use this so you understand why something was flagged.
--min-confidence 50Only show functions with a confidence score of 50 or above. Default is 40. Lower = more results (more noise). Higher = fewer results (more certainty). We recommend 75+ for anything you'd consider deleting.
--limit 100How many results to show in the terminal. Default is 50. Use --limit 500 to see the full list. Results are always sorted highest confidence first.
--json-outputInstead of a terminal table, outputs the full results as a JSON object to stdout. Pipe this into a file with > report.json and use it to feed dashboards, CI systems, or Slack bots.
--no-gitSkips the git history analysis entirely. Faster, but the scores will be lower and less accurate since git age and author signals won't be applied. Use this on repos with no git history.
Tip: Run this first, always. It's completely safe — nothing is modified. Use it to get a feel for what the tool finds in your codebase before touching anything.
python3 report.py [path] Generate a full interactive HTML report — open it in your browser ›
python3 report.py ./my-project --output dead_code_report.html
This generates a full HTML dashboard you can open in any browser. It contains every dead code candidate in a filterable, searchable table — with confidence bars, verdicts, language badges, and reasons. Perfect for sharing with your team before deciding what to clean up.

After running the command, it prints the path to the HTML file and a command to open it. On Mac, run open dead_code_report.html — your browser opens immediately with the full interactive report. The file is fully self-contained — no internet required, no server needed.
--output report.htmlWhere to save the HTML file. Defaults to dead_code_report.html in the current directory. You can give it any path — e.g. ~/Desktop/ngage_dead_code.html.
--no-gitSkip git analysis. Faster but less accurate scores. The report will still show all results, just with lower confidence values.
What the report shows: A filter bar (All / Safe to delete / Review first / Needs runtime data), a language dropdown to filter by Dart, Python, etc., a search box to find specific functions or files, and a full table with confidence bars, verdicts, file paths with line numbers, and reasons. Share the HTML file with your team — they can open it without installing anything.
To open on Mac: open dead_code_report.html — opens in your default browser instantly.
python3 clean.py [path] --dry-run Preview exactly what would be deleted — zero changes made ›
python3 clean.py ./my-project --dry-run --min-confidence 85
The dry run is a completely safe preview. It runs the full scan and scoring pipeline, shows you exactly which functions would be deleted and why, and then stops. No files are touched, no branches are created, no git commands are run. It's identical to the real clean except nothing happens at the end.

Always run this first before running the actual clean. Review the list carefully. If you see a function that looks like it might still be needed, either increase --min-confidence or manually check it in your editor before proceeding.
--min-confidence 85The confidence threshold for the dry run preview. Functions below this score won't be shown. We recommend 85 for the first time — it gives you a shorter, higher-confidence list that's easier to review.
--limit 20How many candidates to show in the dry run table. Useful to limit the preview to just the most confident results.
--lang dartOnly preview dead code in a specific language. Useful to scope the preview to one part of your stack — e.g. just Dart files in a Flutter project.
What to look for in the dry run: Check for any function whose name suggests it might still be in active use (like _handleSubmit or anything with "current", "active", or "main" in the name). These are the ones to manually verify in your editor before deleting.
Watch out for: Private methods starting with _ in Dart/Flutter that are called only within the same file via button callbacks. The static analyzer can sometimes miss these. When in doubt, open the file and search for the function name before deleting.
python3 clean.py [path] Full auto-clean — delete, test, commit, open PR ›
python3 clean.py ./my-project --min-confidence 85 --limit 20
This is the full pipeline command. It scans your project, scores every function, shows you the list of candidates, asks for your confirmation, then — if you say yes — creates a new git branch, surgically deletes each flagged function from your source files, runs your full test suite, commits the changes, and optionally opens a GitHub PR.

Your main branch (prod, main, master) is never touched. Everything happens on an isolated branch. If tests fail at any point, the branch is deleted and your codebase is restored to exactly how it was.
--min-confidence 85Only delete functions scoring 85% or above. This is the most important flag. Higher = safer. We recommend starting at 85 for your first run. After you're comfortable with the results, you can lower to 75 to catch more.
--limit 20Maximum number of functions to delete in a single run. Default is 50. We recommend starting with 10–20 on your first real run so the PR is small and easy to review.
--lang dartOnly clean files of a specific language. Useful when you want to scope cleanup to one part of your codebase — e.g. run --lang dart for Flutter files only.
--no-prDeletes the code and commits it to the new branch but does NOT push to GitHub or open a PR. The branch stays local on your machine. Use this to review the changes yourself before pushing.
--no-testsSkips running the test suite. The code is deleted and committed without verification. Only use this if you don't have a test suite or if Flutter is not installed on the machine running the tool.
What happens step by step: (1) Scan runs. (2) Results shown. (3) You confirm with y/n. (4) New branch created. (5) Functions deleted. (6) Tests run. (7) If tests pass → committed. (8) You choose to push and open a PR or keep it local.
Before running: Make sure your working tree is clean — commit or stash any uncommitted changes first. The tool checks for this and will refuse to run if there are pending changes, to avoid any risk of losing your work.
// git_workflow

What happens
to your git.

This is the part most people want to understand clearly. Here's exactly what the tool does to your git history, step by step — and how to merge, review, or roll back.

1
Your main branch is untouched
The tool checks which branch you're on (e.g. prod, main, or master) and records it. Every subsequent operation happens on a new isolated branch. Your current branch is never modified in any way.
✓ Zero risk to main branch
2
A new cleanup branch is created automatically
The tool creates a new branch named with today's date — for example deadcode/cleanup-2026-05-24. This is where all deletions happen. If a branch with that name already exists (e.g. you ran it twice today), it adds a timestamp suffix like deadcode/cleanup-2026-05-24-1430.
git checkout -b deadcode/cleanup-2026-05-24
3
Functions are deleted and committed
Each flagged function is removed from its source file using precise AST byte ranges. Once all deletions are done and tests pass, all modified files are staged and committed in a single commit with a detailed message listing every deleted function, its file location, and its confidence score.
git add -A && git commit -m "chore: remove 20 dead functions"
4
Review the changes on the new branch
Before merging anything, switch to the cleanup branch and look at what changed. Use your editor or run the git diff command to see every deletion side by side. This is your chance to verify everything looks right.
cd /path/to/your/project

git checkout deadcode/cleanup-2026-05-24

git diff main deadcode/cleanup-2026-05-24
You can also open the project in your editor on this branch and run the app manually to verify everything works as expected.
5
If everything looks good — merge it
Switch back to your main branch and merge the cleanup branch in. The deletions are now part of your main codebase.
git checkout prod

git merge deadcode/cleanup-2026-05-24
Alternatively, if you pushed the branch and opened a PR on GitHub, just click Merge pull request in the GitHub UI — same result.
✓ Clean history, one commit per cleanup run
6
If something looks wrong — roll back instantly
If you review the diff and decide you don't want the changes, just go back to your main branch and delete the cleanup branch. Your main branch was never touched, so nothing has changed.
git checkout prod

git branch -D deadcode/cleanup-2026-05-24
That's it. One command and it's as if the tool was never run. No traces left, nothing broken.
⚠ Roll back is instant and complete
7
Auto-rollback if tests fail
If the tool runs your test suite and any test fails, it automatically rolls back — deletes the cleanup branch and restores your original branch — without you having to do anything. You'll see an error message explaining which test failed. Nothing is ever committed if tests don't pass.
✓ Automatic — no manual rollback needed

// confidence_scoring

How the score
is calculated.

Every function gets a 0–100 confidence score built from five independent signals. The higher the score, the more certain we are that the function is genuinely dead and safe to remove.

Pts
Signal
How it works
Weight
45
Call graph
The tool builds a complete map of which functions call which across your entire codebase. If a function has zero inbound calls from non-test, non-entry-point code, it scores the full 45 points. 1 caller = 20pts. 2–3 callers = 8pts. This is the single strongest signal.
20
Git age
How long ago was this file last committed? A function in a file untouched for 2+ years scores 20pts. 1–2 years = 15pts. 6–12 months = 10pts. 3–6 months = 5pts. Recent activity scores 0 — recently edited code is rarely dead.
15
Author count
How many unique authors have ever committed to this file? Single-author files score 15pts — they often contain experimental or personal code that wasn't cleaned up. 2 authors = 7pts. 3+ authors = 0pts (actively maintained by the team).
10
Recursive dead
If all of a function's callers are themselves flagged as dead, the function earns a bonus 10pts. This catches entire dead sub-trees — a cluster of functions that only call each other but are never reached from active code.
10
Commit count
Files with only 1–2 total commits were almost never revisited after being written — a strong signal of experimental or abandoned code. Low commit count = 7pts bonus on top of the age signal.
The three verdicts
Safe to delete
80–100
Multiple strong signals all agree. No callers found, old git history, single author. These are the functions you can delete with high confidence. The auto-clean command only targets these by default.
Auto-deleted by clean.py
Review first
50–79
Likely dead but the signals aren't fully aligned. Maybe the git history is recent or there's more than one author. Open the file and manually verify before deleting. Use --min-confidence 50 to include these in a clean run.
Manual check recommended
Needs runtime data
40–49
Static analysis is uncertain — the function may be called dynamically via reflection, dependency injection, or a string-based dispatch pattern that the AST parser can't trace. Don't delete without adding coverage tracing first.
Add runtime coverage first
// vs_others

Every other tool
stops at a list.

SonarQube, Vulture, ESLint, ts-prune — they all detect dead code. None of them delete it, test it, and open a PR for you.

Existing tools The old way
✗Static analysis only — no git intelligence. A function untouched for 3 years looks the same as one written yesterday.
✗High false positive rate — teams learn to ignore the warnings entirely after the first few wrong ones.
✗One language per tool. You need Vulture for Python, ts-prune for TypeScript, deadcode for Go — all configured separately.
✗Outputs a list of warnings. You manually delete each one, one by one, hoping nothing breaks.
✗No test verification. If you delete the wrong thing, you find out at runtime in production.
✗No git branch safety. You're editing directly on your working branch.
✗No PR. You do the commit, the description, the review request yourself.
Dead Code Archaeologist The new way
✓Static analysis + git history combined. Age, author count, and commit frequency all factor into the score.
✓Multi-signal confidence score dramatically reduces false positives. Only flags what multiple signals agree on.
✓9 languages in one tool. Python, Dart, JS, TS, Go, Java, Rust, Kotlin, Ruby, Swift — one command covers all of them.
✓Goes all the way to deletion and a merged PR. Scan to clean in one command.
✓Runs your full test suite before committing. Nothing lands unless tests pass.
✓Isolated git branch — your main branch is never touched. Roll back in one command.
✓Auto-opens a GitHub PR with full per-function documentation.
